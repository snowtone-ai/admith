## 3. 技術アーキテクチャ（Phase 0 確定設計）

### 3.1 エージェント・ロール（4者）

| ロール | 担当 | 目的関数 | 主要パラメータ |
|---|---|---|---|
| **Seller Agent** | 食品製造業者 | 廃棄コスト最小化 or 売却益最大化 | 副産物種類・量・排出時間・廃棄単価 |
| **Buyer Agent** | 飼料・肥料・アップサイクル業者 | 調達コスト最適化 | 必要スペック・最大受入量・購入価格 |
| **Logistics Agent** | 物流業者 | 車両稼働率向上・空車回送削減 | 車両位置・冷凍設備・既存ルート重複度 |
| **Mediator Agent** | Admith | 全体最適（社会的厚生最大化）・不正検知 | 交渉収束性・価値創造額算出 |

### 3.2 交渉プロトコル：Extended Contract Net Protocol

FIPA ACLのCNPを4者間同時交渉に拡張。

```
Step 1: Call for Proposal (CFP)
  Seller Agentが排出予定リソースをブロードキャスト

Step 2: Multilateral Bidding
  Buyer候補が「購入条件」、Logistics候補が「輸送条件」をMediatorに提出

Step 3: Cross-Matching
  価値創造額 = (廃棄単価 + 購入単価) - 輸送単価 で一次選別

Step 4: Iterative Negotiation
  選択グループ内でオファー ⇔ カウンタープロポーザルを繰り返す
  Gemini APIが各エージェントの推論エンジンとして機能

Step 5: Agreement / Failure
  全員がAgreement Zoneに収束した時点で契約成立
```

### 3.3 サーキット・ブレーカー（デッドロック対策）

| 機能 | 設計詳細 |
|---|---|
| TTL制約 | 交渉開始から30秒以内に合意なければ強制終了・次点候補へ移行 |
| Monotonicity Enforcement | 各ステップで前回より相手に有利な条件を提示することを強制（収束保証） |
| Human-in-the-loop | 異常価格乖離・論理矛盾検知時に取引凍結・管理者通知 |

### 3.4 推論コスト（Tier型設計）

```
Tier 1: ルールベースマッチング（コスト≒0） → 全取引の80%
Tier 2: Gemini Flash系                      → 条件交渉が必要な15%
Tier 3: Pro系LLM                            → 複雑な多者間交渉の5%
```

**1交渉あたりの実測コスト（Gemini Flash系・作成時点）:**
- 平均800トークン/メッセージ × 12往復（4者 × 3ラウンド）= 約10,000トークン
- **API費用: 約$0.008〜0.014（≒1.2〜2.1円）/交渉**（2026-05-21 Gemini 3.1 Flash-Lite/2.5 Flash標準単価、入力:出力=8:2〜5:5、1USD=150円換算。Context Cachingで追加削減余地）

## 5. 法規制マップ（日本）

### 5.1 AIエージェントの契約能力（現行法の解釈）

- **現状:** 日本法上AIエージェントに権利能力なし。「代理人」として所有者企業に帰属
- **電子署名法:** エージェントの行動が事前設定のルール範囲内であれば「本人の意思に基づく署名」として有効と解釈可能
- **実装要件:** エージェントの権限範囲を厳密に設定できる機能 + 契約成立タイミングの利用規約明記

### 5.2 Phase 0 に必要な遵守事項

| 法規 | Admithへの適用 | 対応方針 |
|---|---|---|
| **廃棄物処理法** | Admith自身は運搬・処分しないため処理業許可不要。ただしマニフェスト監視義務 | デジタルマニフェスト連携APIを実装。排出〜最終処分をトレーサビリティ台帳で可視化 |
| **食品リサイクル法** | 100トン/年以上の多量発生事業者に年次報告義務（毎年6月末）。飼料化→肥料化の優先順位法定 | エージェントの目的関数に「法的コンプライアンススコア」を組み込み。飼料化マッチング優先 |
| **貨物利用運搬事業法** | 物流仲介時の運賃受取方法によっては登録必要 | Admithは「情報マッチング」に徹し、運送契約は荷主-配送業者直接締結形式 |

### 5.3 意図データ（Phase 2向け準備）

| | 日本・個人情報保護法 | GDPR | Admithのデフォルト |
|---|---|---|---|
| 適用基準 | 容易に照合し個人を特定できるもの | 識別可能な個人に関する情報 | **より厳しいGDPRを採用** |
| 匿名化 | 匿名加工情報制度あり | 仮名化・匿名化の厳格要件 | **差分プライバシー（Differential Privacy）を実装** |

## 6. 競合分析（数値根拠あり）

### Too Good To Go（最大脅威）

| 指標 | 数値 |
|---|---|
| 年間売上 | 約€6.1億（EBITDA €1,060万の黒字） |
| 登録ユーザー | 1.2億人 |
| 日本提供 | 提供あり |
| B2B展開予想 | 未確認仮説。確定ロードマップとして使わない |

**防衛:** TGTGはB2Cに最適化。動的価格交渉・マニフェスト管理・複雑な物流知見なし。Admithはこれらをエージェントで自動化し、B2B川上に特化。

### 大手物流（ヤマト・佐川）

- AIによる配送量予測・マッチングは競争リスク。ただし生産性20%向上実績は未再検証値として扱う
- **防衛:** オープン・プロトコルとして大手物流エージェントさえ参加者として取り込む

### Big Tech（Google/Microsoft）

- 汎用エージェント基盤の進化は脅威。ただし汎用エージェント取引プロトコル投入時期は未確認仮説
- **防衛:** 「おからの含水率と腐敗速度の関係」等ドメイン固有パラメータを理解するドメイン特化型インフラ層で差別化

### 参入戦略の拡張（新規確定）

| モデル | アナロジー | 適用場面 |
|---|---|---|
| マーケットプレイス型 | Amazon | Cat.5 新市場 |
| **インフラ提供型（新規追加）** | **Stripe** | **Cat.4 既存市場への参入** |

インフラ提供型の具体例: TABETE・物流会社に「エージェント交渉プロトコルAPI」を提供 → デファクトスタンダード化（TCP/IP戦略）。v1のプロトコル層オープン化方針と整合。

## 7. 研究データ CVE（コンビネーション・バリュー・エンジン）設計

### 技術スタック

```
Vector Embeddings（データセットのメタ記述をベクトル化）
+ Cosine Similarity（補完的なデータを検索）
+ Knowledge Graph（変数間の因果関係を定義）
```

### プライバシー保護3層

| 技術 | 役割 | 実装状況 |
|---|---|---|
| Federated Learning | 生データを集めずモデル訓練。更新済み重みのみを統合 | 現技術で実装可能 |
| Secure MPC | 互いのデータを隠したまま「結合時の相関値」を共同計算 | 現技術で実装可能 |
| ZKP | データの統計的性質が正しいことをデータ開示なしで証明 | 実装コスト高。ヘルスケア分野での先行事例あり |

### パイロット顧客候補

- 日本の大学TLO（技術移転オフィス）：休眠特許・未整理実験データが膨大
- ResearchGate / JSPS KAKENHI：論文紐付きデータセットおよび助成金情報との連携
